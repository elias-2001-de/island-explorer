Software Engineering Modelle - Sprint 3

Auf dieser Seite sind die relevanten Software Engineering Modelle zum Ende von Sprint 3 (Sonntag 02.07.2023) hinterlegt. Die Modelle zeigen sowohl den derzeitigen Stand des Projektes auf als auch den geplanten, verbesserten Stand, der in Sprint 4 als Teil des Refactoring durchgeführt wird.

Verwendete Design Patterns

Im Projekt wurden mindestens 2 Program Design Patterns sinnvoll verwendet:

Pattern 1: Singleton

Verwendet in: backend/src/tables/<...>Table.ts

Zweck: In allen Datenbank-Klassen haben wir das Singleton Pattern verwendet. Der Constructor wurde auf private gestellt und eine statische Instanz der Klasse wird in ihr selbst gespeichert. Dies verhindert, dass von einer Datenbank-Tabelle zwei Modelle über Sequelize erstellt werden können. Da es stets nur eine Tabelle desselben Namens/Typs gibt, ist es sinnvoll, auch nur ein Modell davon im Code erstellen zu können.

Pattern 2: Strategy

Verwendet in: backend/src/controllers/PaymentController.ts und backend/src/payment

Zweck: Der PaymentController stellt den Kontext des Strategy-Patterns dar, während die abstrakte Klasse PaymentService das Interface für die Zahlungsstrategien (BachelorCardService, HCIPalService, SWPSafeService) bietet. Die Verwendung des Patterns erlaubt es uns, den Prozess von buyTicket zu abstrahieren und nur das tatsächliche Ausführen der Zahlung für jeden Zahlungsanbieter separat implementieren zu müssen.
